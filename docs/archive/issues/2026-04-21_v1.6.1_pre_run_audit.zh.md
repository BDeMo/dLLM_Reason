## 🔴 Critical (修了)

### DDP wrapper attribute access — Finetuner 在 DDP 下完全跑不动

**Symptom**：v1.6.1 启 `--sft_gpus 8` 后第一个 batch 就 `AttributeError: 'DistributedDataParallel' object has no attribute 'noise_input'`。

**根因**：
- `src/dllm_reason/training/finetune.py` 用 `self.model.noise_input(x_0, t)` / `self.model.mask_token_id` / `self.model.device` —— 这些是 LLaDAWrapper 自定义属性
- DDP 包装后 `self.model` 是 `DistributedDataParallel(LLaDAWrapper)`
- `nn.Module.__getattr__` 只 fall-back 到 `_parameters/_buffers/_modules`，**不**自动转发到 `.module`
- 任何非 forward/parameters() 的属性访问都炸

**修复**：在 `t6t7_train.py` 里定义 `TransparentDDP` 子类，override `__getattr__`：
```python
class TransparentDDP(DDP):
    def __getattr__(self, name):
        try: return super().__getattr__(name)
        except AttributeError: return getattr(self.module, name)
model = TransparentDDP(model, device_ids=[local_rank], ...)
```
代价：~10 行；不 touch 共享 `Finetuner` 代码（避免影响其他 caller）。

## 🟡 Medium (改了 / 记录)

### M1. DistributedSampler 不 set_epoch → 跨 re-iter shuffle 同序

`Finetuner.train()` loop 是 step-based 而非 epoch-based。`DataLoader` exhaust 后重 iter，但 `DistributedSampler` 不调 `set_epoch()` 就**每次 re-iter 同样顺序**。

不影响正确性，影响 randomness。**修了**：t6t7_train 在创建 sampler 后立刻 `set_epoch(args.seed)`，至少跨 rank 序不同。彻底修需要给 Finetuner 加 epoch counter，留 v1.7。

### M2. 默认 max_steps 太大可能严重过训

默认 `T6_MAX_STEPS=2000` × `batch_size=4` × `grad_accum=4` × `sft_gpus=8` = **256k samples views**。

数据规模 ~5k SFT pairs（v1.6.1 multi-output mode）→ **~50 epochs**。SFT 标准 2-5 epochs，50 epochs 几乎必过拟合。

**未改默认**（保 backward compat），但**user 应该传 `--t6_max_steps 500`** 在 5k 样本下，或 `--t6_max_steps 200` 在 1.6k 样本下。

### M3. Phase 5 parallel eval 无 GPU 队列

```bash
# 现在的逻辑
for ck in CKPTS:
    g = (g + 1) % EVAL_GPUS  # round-robin
    launch_bg "CUDA_VISIBLE_DEVICES=$g python eval &"
```

如果 `len(CKPTS) > EVAL_GPUS`，多个 eval 会**同时挤同一 GPU**，OOM 风险。当前 use case `CKPTS = {baseline, t6}` + `EVAL_GPUS=2` 安全。

文档说明：**`--eval_gpus` 应 ≥ ckpt 数**。

### M4. Phase 0 download 失败半中场 → Phase 2 用半下数据

`download_models.py`、`download_datasets.py`、`download_qwen.py` 都有 idempotent + check，但若网络中断半 way → 部分文件存在 + 大小不对，下次 run 可能跳过 verify 直接用残 file。

缓解：每个 download script 都 print "verified" 才算成功；v1.6.1 已加。但 user 应**看 Phase 0 输出确认 ✓** 再让后续跑。

## 🔵 Minor (留意 / 不改)

### m1. clean_trace regex 假设 `<SETUP>` 在 `<ANSWER>` 前

如果 teacher 输出顺序乱（先 `<ANSWER>` 再 `<SETUP>`）regex 不匹配 → trace 被弃。实测 Qwen3-8B 99.9% 顺序正确，OK 不修。

### m2. T6 multi-output dedup 仅按字符串严格相等

两条 trace 只差一个空格也算"不同"被都收入。轻微 inflation 训练数据，不影响正确性。

### m3. v16_eval Phase 5 用 `eval` exec backgrounded command

```bash
eval "CUDA_VISIBLE_DEVICES=$g python ... &"
PIDS+=($!)
```

`$!` 在 `eval` 之后捕获最新 background pid。验过 work 但有点 fragile。重写成显式 `bash -c` + 直接 `&` 更稳，但目前 OK。
