## Fixed

### F1 — `scripts/inspect_stage2.py:74` — wrong import path (HIGH)

**Symptom:** `ImportError` at runtime whenever the inspector tries to load
episode details for a prompt.

**Root cause:** `from dllm_reason.episodes.store import EpisodeStore` — the
module `dllm_reason.episodes` does not exist.

**Fix:** Changed to `from dllm_reason.library.episode import EpisodeStore`.

**Robustness additions:**
- Split the bare `except Exception: return []` into separate `ImportError`
  and `Exception` branches, each emitting a `warnings.warn` so the caller
  knows why episode data is missing instead of silently receiving `[]`.

### F2 — `src/dllm_reason/training/correction_train.py:96` — `attention_mask` dropped in backbone forward (HIGH)

**Symptom:** For padded batches, the backbone attends to padding tokens when
generating the "noisy-clean" sequence, producing corrupted logits.  The
resulting error labels (`is_error`) are wrong, poisoning all subsequent BCE
and CE training signals.

**Root cause:** `_generate_noisy_clean(self, x_0, t_val, prompt_mask)` had
no `attention_mask` parameter; `self.base.forward(x_t, t).logits` was called
without it even though the caller (`_step`) already had `attention_mask`.

**Fix:**
- Added `attention_mask: torch.Tensor | None = None` to the signature
  (keyword-only, defaults to `None` for backward compat).
- Forwarded it: `self.base.forward(x_t, t, attention_mask).logits`.
- Updated `_step` to pass `attention_mask` through:
  `self._generate_noisy_clean(x_0, t_val, prompt_mask, attention_mask)`.

### F3 — `src/dllm_reason/training/rl_train.py` — `prompt_mask[0].sum()` hardcodes batch index 0 (MEDIUM)

**Affected locations:**
| Class | Former line | Role |
|---|---|---|
| `DiffuGRPO.train()` | 158 | `gen_length` passed to `DiffusionSampler.sample()` |
| `DiFFPO.train()` | 397 | same |
| `UnmaskingPolicyRL._policy_rollout()` | 792 | `gen_start` used to mask generation region |
| `UnmaskingPolicyRL.train()` | 864 | `gen_length` passed to `_policy_rollout` |

**Symptom:** If a batch contains samples with different prompt lengths,
only sample 0's length is used to compute `gen_length` / `gen_start`.
Other samples are masked from the wrong position: some prompt tokens are
treated as generation tokens (or vice-versa), producing silently wrong
reward slicing and rollout sequences.

**Fix:**
- Added module-level helper `_resolve_prompt_len(prompt_mask)`:
  - Computes `prompt_lens = prompt_mask.sum(dim=-1)` (per-sample).
  - Returns `max` across the batch — guarantees no prompt token is exposed
    as a generation position.
  - Emits `logger.warning` when lengths are heterogeneous so the user can
    investigate the collator.
- Replaced `int(prompt_mask[0].sum().item())` with `_resolve_prompt_len()`
  at all three training-loop sites (DiffuGRPO, DiFFPO, UnmaskingPolicyRL.train).
- `_policy_rollout` masking vectorised:
  ```python
  gen_starts = prompt_mask.sum(dim=-1)                             # (B,)
  positions  = torch.arange(L, device=device).unsqueeze(0).expand(B, -1)
  in_gen_region = positions >= gen_starts.unsqueeze(1)             # (B, L)
  x[in_gen_region] = self.model.mask_token_id
  ```
  Each sample now masks from its own prompt boundary.

### F4 — `src/dllm_reason/inference/validation_ext.py:112` — double softmax (LOW)

**Symptom:** `committed_conf` is written with slightly wrong probability
values, subtly biasing the span-revise confidence threshold.

**Root cause:** `generate_span_revise` computed
`p = F.softmax(logits.double(), dim=-1)` at line 99, then called
`F.softmax(logits.float(), -1)` again at line 112 on the *same* `logits`
to gather committed-token confidences.  A second softmax of already-softmaxed
values produces a flatter (less peaked) distribution.

**Fix:** Replaced the second `F.softmax` call with `p.float()[0]` to reuse
the already-computed probabilities.

### F5 — `src/dllm_reason/training/progressive_train.py:204` — scalar int in `torch.where` (LOW)

**Symptom:** `RuntimeError: expected scalar type Long` on PyTorch < 2.1.

**Root cause:**
```python
x_t = torch.where(is_masked, self.model.mask_token_id, x_0)
```
The second argument is a plain Python `int`.  PyTorch ≥ 2.1 promotes it
automatically; earlier versions do not.

**Fix:** Replaced with `masked_fill`, which is unambiguous about dtype
and idiomatic for this operation:
```python
x_t = x_0.masked_fill(is_masked, self.model.mask_token_id)
```
